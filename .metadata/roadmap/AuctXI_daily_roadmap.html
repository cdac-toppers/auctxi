Section 02

The architecture, as simple as it can be.

A monolith-first Spring Boot core handles the auction domain. Two purpose-built .NET microservices — Payment and Notification — sit beside it. No premature decomposition. No "auth service". No "player service". Just three deliberate runtime units.

Guiding principle — Keep the architecture professional, but don't introduce complexity just to make the project look sophisticated.

Spring Boot Core Responsibilities

Authentication · Authorization · Users · Players · Teams · Auctions · Auction Rules · Bidding · Player Allocation · Reports

.NET Payment Service

Create Razorpay Order · Verify Payment Signature · Payment Status · Transaction Handling

.NET Notification Service

Email notifications · Auction events · Payment confirmation · Bid/winner notifications

Section 04

Four repositories. Not ten.

Documentation lives in /docs inside the backend repo or in Notion. No dedicated docs repo. No per-service repo for microservices that live together.

# AuctXI GitHub Organization auctxi-frontend auctxi-backend auctxi-services ├── payment-service └── notification-service auctxi-infra ├── docker └── jenkins
Section 05

A branch model small enough to actually follow.

Every repo runs the same shape: main is sacred, develop is integration, features branch off and merge back through PRs. Branches are named after Jira tickets — not feelings.

main │ └── develop │ ├── feature/... ├── bugfix/... └── test/...

Naming convention

# Preferred — Jira-driven feature/AUX-12-login-ui feature/AUX-15-jwt-authentication feature/AUX-24-player-crud # If contributor visibility is needed feature/AUX-12-vijay-login-ui

Rule — No one pushes directly to main. Both main and develop are protected. PR review is mandatory before merge.

Section 06

How everyone learns everything.

No one leaves AuctXI as "the frontend guy" or "the backend guy". Four mechanisms force cross-pollination across React, Spring Boot, MySQL, .NET, testing, Docker and Jenkins.

01

Rotating implementation

Every member works on different technologies across different days. You don't get to hide in your comfort zone.

02

Cross-review

You never review your own PR. The reviewer is responsible for understanding the change well enough to explain it the next morning.

03

Pair rotation

High-risk areas — authentication, live bidding, Docker, Jenkins — are built collaboratively, not soloed.

04

Daily knowledge transfer

End of every day: developer explains what they built, reviewer asks questions, team documents decisions.

The loop — Person who writes code → Person who reviews code → Person who tests code → Entire team understands the feature.

Section 09

The daily rhythm.

Same shape, every day, for nine days. Predictability is what makes the rotation survivable.

Morning — 15 minutes

# Jira stand-up What did I complete? What will I do today? What is blocking me? # Move ticket TO DO → IN PROGRESS

During development

Jira Story → Create Branch → Develop → Write Unit Tests → Run Tests → Commit → Push → PR

Evening

PR → Cross Review → Integration Test → Merge → develop

Then a 20-minute knowledge transfer: "Today I implemented X. This is how it works." The reviewer should be able to explain it the next morning.

Section 11

The professional workflow, end to end.

From requirement to deployment — the path every feature walks. This is what AuctXI looks like as a system, not a list of tasks.

Requirements ↓ Jira Epic ↓ User Story + Acceptance Criteria ↓ Task Assignment ↓ Feature Branch ↓ Development ↓ Unit Testing ↓ Commit + Push ↓ Pull Request ↓ Cross Code Review ↓ Integration Testing ↓ Merge to Develop ↓ Docker ↓ Jenkins CI/CD ↓ System Testing ↓ Release to Main ↓ v1.0.0 ↓ Deployment

Before Day 1 — Don't start coding authentication yet. The next move is to translate this blueprint into a complete, implementation-ready Jira backlog: Epic → User Story → Acceptance Criteria → Technical Tasks → Unit Tests → Integration Tests → Priority → Story Points → Dependencies → Assigned Member → Git Branch Name. Then Day 1 opens with a board that's ready to burn down.
