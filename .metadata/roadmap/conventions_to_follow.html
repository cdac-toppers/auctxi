Team Development Convention · v1.0

The rules of the build.

Before the four of you touch a single line of AuctXI, agree on the contract. These rules prevent naming chaos, merge conflicts, duplicate code, and broken architectures. Keep them practical. Enforce them in PR reviews.

33
Development Rules
04
Engineers Aligned
01
Source of Truth

The Golden Rule — Never code directly on main or develop. Every change starts from an updated develop, lives on a feature branch, and survives a cross-review before it merges back.

Section 01 — 04

Git, Branching & Workflow.

If Jira doesn't know about it, it didn't happen. If it's not on a feature branch, it doesn't ship.

01
Golden Rule: Never Code Directly on main or develop

Every change starts from an updated develop.

git checkout develop git pull origin develop git checkout -b feature/AUX-21-player-management
Jira Ticket ↓ Feature Branch ↓ Code + Tests ↓ Commit ↓ Push ↓ Pull Request ↓ Code Review ↓ Merge into develop

main = stable/release code only.

02
Branch Naming Convention
Use
feature/AUX-21-player-management feature/AUX-35-live-bidding bugfix/AUX-42-invalid-bid-validation hotfix/AUX-51-payment-failure test/AUX-60-auction-integration-tests docs/AUX-70-update-api-documentation
Avoid
vijay-branch new test final final2 working latest

The Jira ID tells everyone exactly why the branch exists.

Section 03 — 04

Commits & Pull Requests.

03
Commit Message Convention

Follow simple conventional commits.

feat: add player registration API fix: prevent bids exceeding team purse test: add unit tests for BidService docs: update auction API documentation refactor: extract bid validation logic chore: configure Docker Compose
Avoid
changes done final update working now bug fixed

Commit small logical changes instead of one massive end-of-day commit.

04
Pull Request Convention

Every PR should contain structured information.

Title: [AUX-21] Add Player Management Description: Implemented CRUD operations for player management. Changes: - Added Player entity - Added PlayerRepository - Added PlayerService - Added PlayerController - Added validation - Added unit tests Testing: - Unit tests passed - Integration tests passed - API tested through Postman Jira: AUX-21

Rule — Another team member must review it. Never approve and merge your own PR.

Section 05 — 08

Java & Spring Boot Architecture.

05
Java Naming Conventions

Classes — PascalCase

PlayerController PlayerService PlayerRepository Player PlayerRequestDto

Variables and Methods — camelCase

playerName basePrice teamId getPlayerById() placeBid() createAuction()

Constants — UPPER_SNAKE_CASE

MAX_SQUAD_SIZE DEFAULT_BID_INCREMENT MAX_TEAM_PURSE

Packages — lowercase

com.auctxi.controller com.auctxi.service com.auctxi.repository com.auctxi.entity
06
Spring Boot Package Structure

Everyone follows the same structure.

com.auctxi │ ├── config ├── controller ├── dto ├── entity ├── exception ├── mapper ├── repository ├── security ├── service └── util
Don't randomly create
helpers stuff common2 temp new misc

Create a package only when it has a clear responsibility.

07
Controller → Service → Repository
Request ↓ Controller ↓ Service ↓ Repository ↓ Database

Controller: HTTP requests, request validation, HTTP responses. Not business logic.

Service: Business rules, bid validation, auction logic, player allocation.

Repository: Database access.

For example, don't put complex bid logic inside BidController.

08
Use DTOs for APIs

Avoid exposing database entities directly.

Request: React ↓ PlayerRequestDto ↓ Controller ↓ Service ↓ Player Entity ↓ Database Response: Database ↓ Entity ↓ PlayerResponseDto ↓ React

This keeps your API separate from your database model.

Section 09 — 11

API Design & Error Handling.

09
REST API Naming Convention

Use nouns, not verbs. HTTP method already describes the operation.

Good
GET /api/players GET /api/players/{id} POST /api/players PUT /api/players/{id} DELETE /api/players/{id} GET /api/auctions POST /api/auctions GET /api/auctions/{id}/bids POST /api/auctions/{id}/bids
Avoid
/getPlayers /addPlayer /deletePlayer /doBid /getAllTeams
10
Standard API Responses

Keep error responses consistent. Don't have every controller returning completely different error formats.

{ "timestamp": "2026-07-20T10:30:00", "status": 400, "error": "Bad Request", "message": "Bid amount exceeds available team purse", "path": "/api/auctions/10/bids" }
11
Centralized Exception Handling

Don't write repetitive try-catch blocks in every controller. Use Spring's @RestControllerAdvice with @ExceptionHandler.

Create meaningful exceptions such as:

PlayerNotFoundException TeamNotFoundException AuctionNotFoundException InvalidBidException InsufficientPurseException UnauthorizedAuctionAccessException
Section 12 — 13

Data Types & Validation.

12
Validation at Boundaries

Validate data at the system boundary.

@NotBlank private String playerName; @Positive private BigDecimal basePrice;

Also validate business rules inside services. For bidding:

Auction must be LIVE ↓ Player must be AVAILABLE ↓ Team must be participating ↓ Bid > Current Bid ↓ Bid follows minimum increment ↓ Team has sufficient purse ↓ Team has available squad slot

Never trust data coming from the frontend.

13
Use Correct Data Types (Especially for Money)
Don't use
float price; double bidAmount;
Use
BigDecimal bidAmount; BigDecimal teamPurse; BigDecimal basePrice;

Similarly, use proper date/time types rather than storing dates as strings.

Section 25 — 26

Database Standards.

25
Database Convention

Use consistent names.

users players teams auctions bids payments

Columns

player_id team_id auction_id created_at updated_at

Define Primary keys, Foreign keys, Unique constraints, and NOT NULL constraints. Don't rely entirely on Java validation. Important data integrity rules should also exist at the database level where appropriate.

26
Database Changes Must Be Communicated

Don't silently modify the schema locally and expect everyone else to figure it out.

Developer ↓ Schema change ↓ Commit migration ↓ Inform team ↓ PR Review

For your short project, you can start with shared SQL migration scripts. If you have enough time, use Flyway with Spring Boot—but don't add it just for the sake of having another tool.

Section 14 — 16

React & Frontend Standards.

14
React Naming Convention

Components

PlayerCard.jsx AuctionCard.jsx LoginForm.jsx BidHistory.jsx

Variables & Functions

playerName currentBid remainingPurse handleLogin() handleBid() fetchPlayers()

Hooks

useAuth() useAuction()
15
React Folder Convention

Avoid one giant components folder with 100 unrelated files.

├── assets/ ├── components/ │ ├── common/ │ ├── forms/ │ └── layout/ ├── pages/ │ ├── admin/ │ ├── manager/ │ └── team-owner/ ├── services/ ├── hooks/ ├── context/ ├── routes/ ├── utils/ └── constants/
16
Don't Call APIs Directly Everywhere

Keep API communication centralized. Components use the service instead of repeatedly writing Axios configuration.

├── authService.js ├── playerService.js ├── teamService.js ├── auctionService.js └── bidService.js
export const getPlayers = () => { return api.get("/players"); };
Section 17 — 18, 27 — 29

Environment, Logging & Security.

17
Environment Variables

Never hard-code: Database passwords, JWT secrets, Razorpay secrets, API keys.

Never commit
.env application-prod.properties secrets.json
Commit templates
VITE_API_URL= DB_URL= DB_USERNAME= DB_PASSWORD= JWT_SECRET= RAZORPAY_KEY_ID= RAZORPAY_KEY_SECRET=

Each developer maintains their own local values.

18
Logging Convention
Do not use random prints
System.out.println("here"); System.out.println("working");
Use proper logging
log.info() log.warn() log.error() log.debug()
log.info("Auction {} started by manager {}", auctionId, managerId);

But never log: Passwords, JWT tokens, Razorpay secrets, Sensitive payment information.

27
Payment Rules

Never trust payment success from React. The backend must verify the transaction.

React ↓ Payment Microservice ↓ Razorpay ↓ Payment Verification ↓ Server confirms payment ↓ Database updated
28
Security Rules

All four developers must remember:

- Hash passwords - Never store plain-text passwords - Never expose JWT secrets - Validate JWT server-side - Implement role-based authorization - Validate all input - Never trust frontend validation alone - Don't expose stack traces to users - Don't commit secrets
29
Dependency Rule

Before adding a new npm, Maven, or NuGet package, ask: Do we actually need it?

Don't add libraries just because a tutorial uses them. The fewer unnecessary dependencies, the easier your project is to maintain.

Section 22 — 24

Testing Standards.

22
Unit Testing Convention

Focus unit tests primarily on business logic. Example: BidServiceTest

shouldAcceptValidBid() shouldRejectBidWhenAuctionIsClosed() shouldRejectBidBelowMinimumIncrement() shouldRejectBidWhenPurseIsInsufficient() shouldRejectBidWhenSquadLimitReached()

Use the pattern:

Arrange ↓ Act ↓ Assert # OR Given ↓ When ↓ Then
23
Integration Testing Convention

Test complete component interaction. Don't confuse unit tests with integration tests.

HTTP Request ↓ Controller ↓ Service ↓ Repository ↓ Test Database

Examples:

POST /api/players POST /api/auth/login POST /api/auctions/{id}/bids
24
Bug Fix Convention

Every significant bug gets a Jira ticket.

AUX-107 Bid accepted when team purse is insufficient
bugfix/AUX-107-bid-purse-validation

Add a regression test where practical. Then fix the bug. This proves the bug doesn't return later.

Section 19 — 21, 30

Code Quality.

19
Comments Convention

Don't write comments explaining obvious code. Use comments to explain why, not what.

Bad
// Get player Player player = repository.findById(id);
Good
// Reserve the minimum purse required to fill remaining squad slots BigDecimal availableBidAmount = calculateAvailableBidAmount(team);

Prefer self-explanatory method names over excessive comments.

20
Keep Methods Small

Avoid 300-line methods. Break logic into methods. A method should ideally have one clear responsibility.

public void conductAuction() { validateAuctionStatus(); validateTeamEligibility(); validateBidAmount(); saveBid(); updateHighestBid(); }
21
Avoid Duplicate Code

If the same logic appears multiple times, consider extracting it. For example, don't implement purse validation independently in three controllers. Have one clear business method.

But don't over-engineer either.

Rule — Abstract when duplication becomes meaningful, not before.

30
Code Formatting

Everyone should use automatic formatting. Don't waste PR reviews arguing about spaces and indentation.

# React: Prettier + ESLint # Java: IDE formatting consistently # C#: Visual Studio / .NET formatting
Section 33

The Most Important Business Logic Rule.

33
Data Consistency & Concurrency

For critical operations such as bidding and payment, think about data consistency and concurrency.

Imagine:

Team A bids ₹10 lakh Team B bids ₹11 lakh Team C bids ₹12 lakh # at nearly the same time.

Your backend—not React—must decide which bid is valid. Similarly:

Current Purse = ₹50 lakh Two simultaneous requests ↓ Both try spending ₹30 lakh

You cannot allow both to succeed if that violates the purse.

The Rule — Frontend displays the state. Backend is the source of truth. Database protects persistent data.

This should be one of your core architectural principles.

Section 31 — 32

Definition of Done & Sync.

31
Definition of Done

A feature is not done when "It works on my laptop." A Jira story is done when:

✓ Implementation complete ✓ Naming conventions followed ✓ Validation implemented ✓ Error handling implemented ✓ Logging added where needed ✓ Unit tests written ✓ Integration tests written where applicable ✓ Tests passing ✓ API documented ✓ No secrets committed ✓ PR created ✓ Another member reviewed PR ✓ Merged into develop
32
Daily Sync Rule

Every morning, 10–15 minutes. Each person says:

1. What I completed yesterday. 2. What I'm working on today. 3. What is blocking me.

Every evening:

PR Review + Integration Check + 15-min Knowledge Transfer

The knowledge-transfer part is important for your team because everyone wants to understand the entire project.

The Final Word

AuctXI Development 10 Commandments.

If you want a short version that all four of you can keep pinned to your monitors, memorize these.

01
Never push directly to main or develop.
02
Every code change must be associated with a Jira ticket and branch.
03
Every feature goes through a Pull Request and cross-review.
04
Follow consistent naming and folder conventions.
05
Controller handles HTTP; Service handles business logic; Repository handles data.
06
Never trust frontend validation—validate again on the backend.
07
Never commit passwords, API keys, JWT secrets, or .env files.
08
Every important business rule must have tests.
09
Critical bugs should get regression tests.
10
If you write it, someone else reviews it; if someone else writes it, learn how it works.
