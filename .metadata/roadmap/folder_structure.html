Architecture & Folder Blueprint

Four repositories. No chaos.

Clean enough to be professional, without creating unnecessary overhead. This blueprint defines exactly where every file lives across the AuctXI ecosystem—Frontend, Backend, Microservices, and Infrastructure.

Section 01

The 4-Repository Architecture.

Each runtime unit gets its own Git history and branches. No permanent branches for Vijay, Akram, Rahil, or Prajwal—only branches based on Jira tickets.

Rule — Don't create permanent branches for team members. Create branches based on the Jira ticket being worked on (e.g., feature/AUX-13-jwt-auth).

Section 02

Frontend Repository.

The React application. Kept strictly separated by domain (auth, admin, manager, teamOwner) to prevent a giant soup of components.

auctxi-frontend
auctxi-frontend/ │ ├── public/ │ ├── src/ │ │ │ ├── assets/ │ │ ├── images/ │ │ ├── icons/ │ │ └── logos/ │ │ │ ├── components/ │ │ ├── common/ (Button.jsx, Loader.jsx, Modal.jsx) │ │ ├── layout/ (Navbar.jsx, Sidebar.jsx, Footer.jsx) │ │ ├── auction/ │ │ ├── player/ │ │ └── team/ │ │ │ ├── pages/ │ │ ├── auth/ (LoginPage.jsx, RegisterPage.jsx) │ │ ├── admin/ (AdminDashboard.jsx, AuctionsPage.jsx, ...) │ │ ├── manager/ (ManagerDashboard.jsx, LiveAuctionPage.jsx, ...) │ │ ├── teamOwner/ (TeamOwnerDashboard.jsx, MyTeamPage.jsx, ...) │ │ └── public/ (HomePage.jsx, AuctionDetailsPage.jsx, ...) │ │ │ ├── services/ (api.js, authService.js, playerService.js, ...) │ ├── context/ (AuthContext.jsx) │ ├── hooks/ (useAuth.js) │ ├── routes/ (AppRoutes.jsx, ProtectedRoute.jsx) │ ├── utils/ (helpers.js) │ ├── constants/ (constants.js) │ ├── styles/ │ ├── App.jsx │ └── main.jsx │ ├── tests/ ├── .env (DO NOT COMMIT) ├── .env.example (COMMIT) ├── .gitignore ├── Dockerfile ├── eslint.config.js ├── package.json ├── vite.config.js └── README.md

Env Vars — .env must never be committed. .env.example should be committed with empty values (e.g., VITE_API_BASE_URL=) so the team knows what keys are required.

Section 05

Infrastructure Repository.

Holds deployment-related configuration. Individual Dockerfiles live with their apps, but the orchestration (Docker Compose) and pipelines (Jenkins) live here.

Section 06

How Everything Connects.

The infrastructure repository wraps the application repositories. The frontend talks to the backend, and the backend talks to the microservices and the database.

Section 07

The Day 1 Strategy.

Do not create this entire folder tree manually upfront. Create only the structural foundation. The structure should grow naturally with the implementation.

Avoid the beginner mistake — Do not create a massive "professional-looking" folder structure filled with empty folders and placeholder files.

What actually gets created on Day 1

Day 1 Minimal Structure
# Frontend src/ ├── components/ ├── pages/ ├── services/ (api.js only) └── routes/ # Backend com.auctxi/ ├── controller/ (HealthController.java) ├── service/ ├── repository/ ├── entity/ ├── dto/ └── config/ # Services ├── payment-service/ (Basic .NET Web API skeleton) └── notification-service/ (Basic .NET Web API skeleton) # Infra ├── docker/ ├── jenkins/ └── docs/

Then add security/ when you start authentication on Day 2, auction/ components when you start auction development on Day 4, Tests/ as you implement functionality, etc.
